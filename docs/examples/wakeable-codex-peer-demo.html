Agent Peers Example

Wakeable Codex Peers, Without the Fog

This page explains how a paused-looking Codex session can be nudged when a peer message arrives, why it needs a special launcher, and how to run the Google Ads Expert plus CCR Website demo.

The honest version

We did not change every Codex session on your machine. We built a managed way to start Codex so the peer network can nudge that same live session when it has unread mail.

The Problem

Claude Code

Claude can be pushed

When a peer sends Claude a message, Claude Code has a channel that can wake the session and show the message mid-task.

Stock Codex

Codex only hears when it checks

A normal Codex session does not show MCP push notifications to the model. If it is sitting at a prompt, it may not notice peer mail until it calls check_messages.

Our workaround

Start Codex in wakeable mode

The wakeable launcher starts Codex through its app-server. That gives the wake daemon a safe handle to nudge the exact same live thread.

The Parts

1. Peer Message

Another peer sends mail

Claude, Codex, or the shell CLI sends a message to ccr-codex or google-ads-codex.

2. Broker

Local post office

The broker on localhost stores the message in SQLite and leases it to the receiving peer's MCP server.

3. Codex inbox metadata

"You have mail"

Codex's MCP sidecar writes tiny bodyless metadata to disk. It says unread mail exists, but it does not put the message body in the wake signal.

4. Wake daemon

Small loop you run once

The daemon sees unread metadata, checks that the target Codex thread is live and idle, then sends a nudge.

5. App-server thread

Same live Codex session

The nudge is injected into the already-open Codex thread. No restart. No new blank context. No fake resumed clone.

6. Codex checks inbox

Real message arrives

Codex calls check_messages. Only then does it receive the actual peer message body and respond like a normal colleague.

In One Sentence

The launcher gives Codex a live app-server handle; the MCP sidecar leaves a "mail waiting" note; the daemon uses the handle to tell the same Codex thread to check its real inbox.

What Works, What Does Not

Situation Expected result
Codex was started with codexpeer, adspeer, or ccrpeer Wakeable, as long as the terminal is still alive and the wake daemon is running.
A normal codex session is already sitting at a prompt Not wakeable by this system. It can still use the peer network when it manually checks messages.
The Codex process was killed, the Mac slept hard, or the terminal was closed Not wakeable. Start the peer again.
Codex is actively working, waiting on approval, or asking you a question The daemon skips the nudge. It avoids interrupting unsafe states.

First-Time Setup

The simple aliases are in ~/.zshrc, next to your opus alias. Open a new Zed terminal, or run this once in an existing terminal:

source ~/.zshrc

Everyday use is simple: go to any repo and type codexpeer. That launches wakeable Codex in that repo and auto-starts the background wake daemon if it is not already running — you no longer need to run peerwake by hand. Check the daemon any time with peerstatus or codexpeer daemon-status, and stop it with codexpeer daemon-stop.

The underlying helper command is codex-peer. It lives in this repo at /Users/mike/agent-peers-mcp/bin/codex-peer and is also linked into ~/.local/bin. If the link ever disappears, reinstall it with:

cd /Users/mike/agent-peers-mcp
bin/codex-peer install

Troubleshooting

  • I sent a message and nothing woke up. Run peerstatus. The target should say wakeable=yes. Also confirm the background daemon is up with codexpeer daemon-status (any codexpeer launch will auto-start it).
  • The peer name is different. Run peerstatus. If a name was already taken, the broker may have suffixed it.
  • The peer says wakeable=no but I started it with codexpeer. Run peerrepair <name>. This repairs the wake pointer without restarting the Codex context.
  • I am done with a peer. Run peerretire <name>. That removes it from discovery/wake tracking without deleting message history.
  • The receiving Codex is working on something. The daemon skips active sessions. Let it finish or bring it back to idle.
  • I started plain codex. That session can use agent peers, but it is not wakeable. Start with codexpeer.
  • I closed the terminal. The live app-server handle is gone. Start that peer again.
